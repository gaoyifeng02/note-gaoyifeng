# s-pay-mall - 原理架构

---

## 1. DDD 分层架构

项目严格遵循 DDD（领域驱动设计）六层架构，模块间通过接口实现依赖倒置：

```
                    ┌──────────────┐
                    │     app      │  ← 启动层，依赖所有模块
                    └──────┬───────┘
         ┌─────────────────┼──────────────────┐
         ▼                 ▼                  ▼
  ┌──────────────┐  ┌──────────────┐  ┌───────────────────┐
  │   trigger    │  │    domain    │  │  infrastructure    │
  │ (HTTP/Job)   │  │  (核心逻辑)   │  │  (DB/Redis/网关)    │
  └──────┬───────┘  └──────┬───────┘  └────────┬──────────┘
         │                 │    ▲               │
         │                 ▼    │               │
         │           ┌─────────┐│               │
         │           │   api   ││               │
         │           └─────────┘│               │
         │                 ▼    │               │
         │           ┌─────────┐│               │
         │           │  types  │◀───────────────┘
         │           └─────────┘
         └─────────────────────▶ domain（调用领域服务）

    infrastructure 实现 domain 中定义的端口接口（依赖倒置）
```

### 各层职责

| 层 | 模块 | 职责 |
|----|------|------|
| API 层 | s-pay-mall-api | 定义对外服务接口、DTO、Response 契约 |
| Types 层 | s-pay-mall-types | 通用常量、枚举、异常、事件基类、SDK工具 |
| Domain 层 | s-pay-mall-domain | 核心领域逻辑：实体、值对象、聚合根、领域服务、端口接口 |
| Infrastructure 层 | s-pay-mall-infrastructure | 实现 Domain 定义的端口：DAO、Redis、外部网关 |
| Trigger 层 | s-pay-mall-trigger | 外部触发入口：HTTP Controller、定时 Job、事件 Listener |
| App 层 | s-pay-mall-app | Spring Boot 启动类、配置组装、测试入口 |

## 3. 核心流程

### 3.1 下单支付流程（模板方法模式）

`AbstractOrderService.createOrder()` 定义了标准下单流程：

```
createOrder(ShopCartEntity)
    │
    ├─ 1. 查询未支付订单（防重复 + 掉单恢复）
    │   ├─ 存在 PAY_WAIT 状态 → 直接返回原支付链接
    │   └─ 存在 CREATE 状态 → 重新生成支付单
    │
    ├─ 2. 查询商品信息（通过 IProductPort）
    │
    ├─ 3. 构建订单聚合根
    │   ├─ 生成16位随机订单号
    │   └─ 状态初始化为 CREATE
    │
    ├─ 4. 保存订单（doSaveOrder - 子类实现）
    │   └─ MySQL + Redis 双写
    │
    └─ 5. 创建支付单（doPrepayOrder - 子类实现）
        ├─ 调用支付宝 SDK 生成支付表单
        ├─ 更新订单状态为 PAY_WAIT
        └─ 返回 PayOrderEntity
```

`OrderService` 继承 `AbstractOrderService`，实现具体的支付宝支付单生成逻辑。

### 3.2 支付回调流程

```
支付宝异步通知 → AliPayController.payNotify()
    │
    ├─ 1. 验证 trade_status == "TRADE_SUCCESS"
    ├─ 2. 收集所有请求参数
    ├─ 3. RSA2 签名验证（AlipaySignature.rsa256CheckContent）
    └─ 4. 调用 orderService.changeOrderPaySuccess()
        │
        ├─ 更新 DB 订单状态为 PAY_SUCCESS
        └─ 通过 EventBus 发布 PaySuccessMessageEvent
            └─ OrderPaySuccessListener 接收（可扩展：发货、充值、开会员）
```

### 3.3 微信扫码登录流程

```
前端 → GET /api/v1/login/weixin_qrcode_ticket
    │
    └─ LoginService.createQrCodeTicket()
        └─ 调用微信API创建临时带参数二维码 → 返回 ticket

用户扫码 → 微信服务器 → POST /api/v1/weixin/portal/receive
    │
    ├─ 验签（SHA1 签名校验）
    ├─ 解析 XML 消息体
    ├─ 识别 event=SCAN
    └─ WeixinLoginService.saveLoginState(ticket, openid)
        ├─ openidToken Cache 写入 (ticket → openid)
        └─ 发送模板消息通知用户

前端轮询 → GET /api/v1/login/check_login?ticket=xxx
    │
    └─ 从 Cache 查询 → 返回 openid（实际可生成 JWT Token）
```

## 4. 定时任务

| 任务 | 频率 | 说明 |
|------|------|------|
| NoPayNotifyOrderJob | 每 3 秒 | 查询未收到回调的已支付订单，主动查询支付宝交易状态补偿 |
| TimeoutCloseOrderJob | 每 10 分钟 | 查询超时30分钟未支付订单，自动关单 |

## 6. 事件驱动架构

```
OrderRepository.changeOrderPaySuccess()
    │
    ├─ 更新 DB
    └─ EventBus.post(PaySuccessMessage)
            │
            ▼
OrderPaySuccessListener.handleEvent()
    └─ 可扩展处理：发货、充值、开通会员等
```

当前使用 Guava EventBus（进程内），topic 为 `pay_success`。`BaseEvent<EventMessage>` 抽象类提供了统一的事件消息结构（id + timestamp + data），未来可无缝切换到 RocketMQ/Kafka。

## 7. 缓存策略

| 场景 | 缓存 Key | 内容 | 说明 |
|------|----------|------|------|
| 订单创建 | `PayOrder.cacheKey(userId, orderId)` | PayOrder PO 对象 | 下单时写入，DB + Redis 双写 |
| 登录状态 | ticket 字符串 | openid 字符串 | Guava Cache，扫码登录后写入 |

Redis 使用 Redisson 客户端，配置连接池大小10、最小空闲5。
