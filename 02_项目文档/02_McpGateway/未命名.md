  一、选择 SSE 的底层原因

  是的，核心原因就是 MCP 协议规范规定了这种传输方式。 MCP（Model Context Protocol）是 Anthropic 提出的开放标准，它定义了一套 JSON-RPC 2.0 消息格式 +
  传输机制。

  MCP 规范定义了两种传输机制：
  1. stdio — 标准输入输出，适用于本地子进程通信
  2. HTTP with SSE（旧版）/ Streamable HTTP（新版）— 适用于网络通信

  你的项目是一个网络网关，需要让远程客户端（比如 Claude Desktop、Cursor 等 AI 工具）通过网络连接，所以必须用 HTTP 传输方式，而这种方式底层就是依赖 SSE 的。

  二、MCP 完整交互过程

  整体生命周期（3 个阶段）

  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
  │  初始化阶段   │ ──→ │   操作阶段    │ ──→ │   关闭阶段   │
  │ Initialize  │     │  Operation  │     │  Shutdown   │
  └─────────────┘     └─────────────┘     └─────────────┘

  详细交互时序

  以你的项目为例（旧版 HTTP+SSE 传输），完整流程如下：

  客户端 (AI工具/Claude Desktop)                    服务端 (你的 MCP Gateway)

  三、关键概念解读

  1. 为什么是两条通道？

  | 方向        | 方式        | 用途                |
  |-----------|-----------|-------------------|
  | 客户端 → 服务端 | HTTP POST | 发送 JSON-RPC 请求/通知 |
  | 服务端 → 客户端 | SSE 推送    | 返回响应、推送通知         |

  客户端发消息是普通的 HTTP POST，服务端回消息是通过 SSE 流推回去。这种设计的好处是：
  - 服务端可以随时主动推送（不需要客户端轮询）
  - SSE 基于标准 HTTP，兼容性好
  - 不需要 WebSocket 的全双工复杂度

  2. JSON-RPC 消息的三种类型

  你的项目中 McpSchemaVO 定义了这三种类型：

  | 类型               | 特征                  | 示例                                 |
  |------------------|---------------------|------------------------------------|
  | Request（请求）      | 有 id + method，需要响应  | initialize, tools/list, tools/call |
  | Response（响应）     | 有 id + result/error | 对请求的回复                             |
  | Notification（通知） | 有 method，无 id，不需要响应 | notifications/initialized          |

  3. 初始化握手的"三步握手"

  客户端                                    服务端
    │  ── initialize request ──→              │   第1步：客户端告知自己的能力
    │  ←── initialize result ───              │   第2步：服务端回应自己的能力
    │  ── initialized notification ──→        │   第3步：客户端确认，握手完成

  这三步必须按顺序执行，初始化完成前不能发送其他请求。

  4. 你的项目中的策略模式

  你的 SessionMessageHandlerMethodEnum 把 MCP 方法映射到了不同的处理器：

  "initialize"     → InitializeHandler    （初始化握手）
  "tools/list"     → ToolsListHandler     （列出工具）
  "tools/call"     → ToolsCallHandler     （调用工具）
  "resources/list" → ResourcesListHandler （列出资源）

  四、回答你的核心问题

  选择 SSE 的底层原因，是因为要支持 MCP 协议吗？

  是的。 MCP 协议规范要求网络传输使用 HTTP + SSE（或新版 Streamable HTTP），你的项目作为 MCP 网关，必须在传输层实现这套机制。

  所以我要在这里底层实现一套 MCP 协议的连接过程？

  是的。 你需要实现：
  5. SSE 连接建立 — 客户端 GET 请求建立长连接
  6. 会话管理 — 创建/维护/清理 Session
  7. JSON-RPC 消息解析 — 解析三种消息类型
  8. 初始化握手 — 三步握手，能力协商
  9. 方法路由 — 根据 method 分发到不同处理器
  10. 业务处理 — tools、resources 等具体功能的实现

  这就是你项目中各层分工的意义：trigger 层处理 HTTP/SSE，domain 层处理协议逻辑，infrastructure 层持久化配置。
