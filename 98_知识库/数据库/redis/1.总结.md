# 🚀 Redis 核心知识速查手册 (Review Cheat Sheet)

### 一、 缓存三大痛点 (穿透 / 击穿 / 雪崩)

- **缓存穿透 (查无此数据)**
    
    - **现象：** 查根本不存在的Key，全打在DB上。
        
    - **对策：** 1. 缓存空对象 (设短TTL)； 2. **布隆过滤器** (前置拦截，极省内存但有极低误判)； 3. 参数校验。
        
- **缓存击穿 (单点爆破)**
    
    - **现象：** 单个**热点Key**突然过期，瞬间高并发打爆DB。
        
    - **对策：** 1. 互斥锁 (加锁查DB重建缓存，其他人等待)； 2. **逻辑过期** (永不物理过期，查到逻辑过期后异步开线程更新，前端先拿旧数据)。
        
- **缓存雪崩 (大面积塌方)**
    
    - **现象：** 大批Key同时过期，或Redis彻底宕机。
        
    - **对策：** 1. **TTL加随机值** (错峰过期)； 2. 高可用集群 (哨兵/Cluster)兜底； 3. 服务限流降级。
        

### 二、 缓存与数据库一致性

- **核心目标：** 追求**最终一致性** (Cache-Aside 旁路缓存)。
    
- **标准动作：** **先更新数据库，后删除缓存**。(注意是删除，不是更新)。
    
- **异常兜底 (删缓存失败怎么办？)：**
    
    - 优雅解法：**Canal 订阅 MySQL Binlog + MQ 异步重试删除** (业务代码完全解耦)。
        
- **历史债务 (非要先删缓存后写DB)：** 采用**延迟双删** (休眠几百毫秒再删一次，防脏数据回写)。
    

### 三、 数据持久化 (RDB vs AOF)

- **RDB (快照/全景照)：**
    
    - **特点：** 二进制全量文件。
        
    - **优缺：** 恢复极快；但容易丢失最后一次快照后的数据。
        
- **AOF (日志/流水账)：**
    
    - **特点：** 记录每次写命令 (默认 everysec 每秒刷盘)。需定期 BGREWRITEAOF 瘦身。
        
    - **优缺：** 最多只丢1秒数据，极安全；但文件大，恢复慢。
        
- **混合持久化 (Redis 4.0+，生产首选)：**
    
    - **机制：** RDB 前缀全量数据 + AOF 后缀增量命令。
        
    - **优势：** 恢复又快，数据又安全。
        

### 四、 数据清理机制 (过期删除 vs 内存淘汰)

- **过期删除策略 (针对有 TTL 的 Key，时间到了怎么删)：**
    
    - **定期抽查删除** (后台默默清理) + **惰性删除** (用户查到过期才删)。保障CPU和内存的平衡。
        
- **内存淘汰策略 (针对内存上限 maxmemory 满了怎么踢)：**
    
    - **全库踢 (最常用)：** allkeys-lru (踢掉最久未使用的，**生产标配**) / allkeys-lfu (踢掉访问频率最低的，4.0引入)。
        
    - **带TTL踢：** volatile-lru / volatile-ttl 等。
        
    - **摆烂：** noeviction (默认，写报错，只准读)。
        

### 五、 集群演进与主从同步

- **架构三步走：**
    
    1. **主从复制：** 读写分离，解决读并发。无高可用。
        
    2. **哨兵模式 (Sentinel)：** 解决高可用，自动选新主。单机内存和写并发仍受限。
        
    3. **分片集群 (Cluster)：** 16384个哈希槽，海量切片。彻底解决写并发和存储瓶颈 (大厂标配)。
        
- **主从同步机制：**
    
    - **全量同步 (初次连/断太久)：** 主节点 bgsave 发 RDB 快照 + replication buffer 补发快照期间的新命令。
        
    - **增量同步 (网络闪断)：** 比对 Offset (进度条)，去 repl_backlog_buffer (环形缓冲区) 把遗漏的增量命令补齐。
        

### 六、 高阶场景 (哨兵 / 脑裂 / 锁)

- **哨兵机制运转流程：**
    
    - 监控心跳 -> 半数以上认为挂了 (**客观下线**) -> 哨兵内部选出领头羊 -> 挑选最优质的从节点晋升新主 -> 强制切换。
        
- **集群脑裂 (Split-Brain)：**
    
    - **致命危害：** 网络分区导致出现双Master，老Master恢复后降级为Slave会**清空自身数据**，导致断网期间客户端写给老Master的新数据**永久丢失**。
        
    - **对策：** 配置 min-replicas-to-write，强制老Master发现身边没小弟时直接拒绝客户端写入。

### 六、 高阶场景 (接上文分布式锁)

- **分布式锁 (防丢锁与续期)：**
    
    - **核心实现：** SET key value NX PX 30000 (原子加锁+过期时间)。
        
    - **释放锁：** 必须使用 **Lua 脚本** 校验 value（唯一标识），防止误删别人的锁。
        
    - **主从切换丢锁：** 客户端 A 在主节点加锁成功，主节点还没同步给从节点就挂了，从节点升级为主后，客户端 B 也能加锁成功。
        
    - **Redlock (红锁)：** 顺序向过半数（N/2+1）个独立 Redis 节点加锁。虽然安全但运维成本高且有争议，**生产环境推荐 Redisson**。
        
    - **锁续期：** **Watch Dog (看门狗机制)**。开启后台线程，每隔 10s 检查业务是否完成，未完成则自动重置过期时间，防止业务没跑完锁先过期。
        

### 七、 底层数据结构 (为什么快？)

- **基础 5 种：**
    
    1. **String：** 动态字符串 (SDS)，预分配内存，O(1) 获取长度。
        
    2. **List：** 双向链表/压缩列表 -> 3.2版本后统一为 **quicklist**。
        
    3. **Hash：** 压缩列表 (ziplist) -> 哈希表 (hashtable)。
        
    4. **Set：** 整数集合 (intset) -> 哈希表。
        
    5. **ZSet (王牌)：** **跳表 (SkipList)** + 压缩列表。实现 O(logN) 的范围查询。
        
- **高级 4 种：**
    
    1. **Bitmaps：** 位图，适用户打卡、签到统计。
        
    2. **HyperLogLog：** 基数统计，极小空间统计海量 UV (有误差)。
        
    3. **GEO：** 地理位置查询 (附近的人)。
        
    4. **Stream (5.0+)：** 消息队列，支持持久化和多消费组 (类似 Kafka)。
        

### 八、 高性能背后的机制

- **线程模型：** 核心执行路径是 **单线程** (避免了多线程上下文切换和竞争锁)，但 6.0 后引入 **多线程 I/O** 处理网络读写。
    
- **I/O 多路复用：** 基于 epoll 机制，一个线程同时监听成千上万个连接。
    
- **高效内存操作：** 数据全在内存，所有计算不涉及磁盘 I/O。
    

### 九、 生产性能避坑 (Big Key / Hot Key)

- **Big Key (大 Key)：**
    
    - **危害：** 阻塞单线程，集群迁移困难，占用带宽。
        
    - **对策：** 拆分（如 hash_01, hash_02）；异步删除 (UNLINK 替代 DEL)。
        
- **Hot Key (热 Key)：**
    
    - **现象：** 某单点 Key 流量过载（如热搜、秒杀）。
        
    - **对策：** 1. 利用**本地缓存** (如 Caffeine/Guava) 挡一层； 2. 给热 Key 副本化 (Key 加随机后缀分摊到不同 Slot)。
        

### 十、 事务与 Lua

- **事务 (Transaction)：** MULTI (开启) -> EXEC (提交)。不支持回滚 (No Rollback)，仅保证命令顺序执行。
    
- **Lua 脚本：**
    
    - **原子性：** Redis 保证整个脚本执行期间不被其他命令插入。
        
    - **优势：** 减少网络开销，复用逻辑，**是实现原子复合操作的首选**。
