Spring 的事务管理是其核心特性之一，主要通过 **Spring AOP** 实现。它提供了一套统一的编程模型，将开发者从繁琐的底层事务代码（如 JDBC 的 `connection.commit()`）中解放出来。

以下是关于 Spring 事务的特点、用法及失效场景的详细分析：

### 1. Spring 事务的特点
Spring 事务的核心设计目标是：**提供一致的事务编程模型**，无论底层使用的是 JDBC、Hibernate 还是 JPA。

*   **ACID 特性：** 遵循原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）。
*   **传播行为（Propagation）：** 定义了当一个事务方法被另一个事务方法调用时，该如何执行（如：加入现有事务、开启新事务等）。常见的有 `REQUIRED`（默认）、`REQUIRES_NEW`、`NESTED`。
*   **隔离级别（Isolation）：** 对应数据库的四种隔离级别（读未提交、读已提交、可重复读、串行化），防止脏读、不可重复读和幻读。
*   **回滚规则：** 默认情况下，Spring 仅在遇到 **运行时异常（RuntimeException）** 和 **Error** 时回滚，受检查异常（Checked Exception）不会导致回滚。
*   **声明式事务：** 结合 AOP，通过注解即可实现事务，对业务代码无侵入。

---

### 2. Spring 事务如何使用？
主要有两种方式：**声明式事务**（推荐）和 **编程式事务**。

#### (1) 声明式事务（常用）
使用 `@Transactional` 注解。
```java
@Service
public class UserService {
    @Transactional(rollbackFor = Exception.class, propagation = Propagation.REQUIRED)
    public void createUser(User user) {
        userMapper.insert(user);
        // 如果这里抛出异常，事务会自动回滚
        if (someCondition) {
            throw new RuntimeException("操作失败");
        }
    }
}
```

#### (2) 编程式事务（灵活但不常用）
适用于需要精确控制事务范围的场景。使用 `TransactionTemplate`。
```java
@Autowired
private TransactionTemplate transactionTemplate;

public void update() {
    transactionTemplate.execute(status -> {
        // 业务逻辑
        return someResult;
    });
}
```

### 3. Spring 事务在什么情况下会失效？
这是面试和实际开发中最常遇到的“坑”，通常由 Spring AOP 的实现原理（动态代理）引起：

1.  **非 Public 方法：**
    *   `@Transactional` 只能应用在 `public` 方法上。如果写在 `private`、`protected` 或 `default` 方法上，事务不会生效且不会报错。
2.  **内部自调用（Self-invocation）：**
    *   **现象：** 同一个类中，方法 A（无事务）调用方法 B（有事务）。
    *   **原因：** Spring 事务是基于代理对象的。在类内部直接调用 `this.B()` 绕过了代理对象，导致事务增强逻辑失效。
    *   **解决：** 注入自己（自我注入）或将方法拆分到不同的 Service 类中。
3.  **异常被“吞”了：**
    *   如果在代码中使用了 `try-catch` 捕获了异常且没有重新抛出，Spring 无法感知到异常，事务会正常提交。
4.  **异常类型不匹配：**
    *   默认只回滚 `RuntimeException`。如果抛出的是 `IOException` 等受检查异常，除非显式配置 `@Transactional(rollbackFor = Exception.class)`，否则事务不回滚。
5.  **数据库引擎不支持：**
    *   如果 MySQL 使用的是 `MyISAM` 存储引擎（不支持事务），那么 Spring 事务自然无效。必须使用 `InnoDB`。
6.  **Bean 没有被 Spring 管理：**
    *   如果类没有加 `@Service` 或 `@Component`，Spring 就不会为其创建代理对象，事务自然无效。
7.  **方法被 `final` 或 `static` 修饰：**
    *   Spring AOP 无法拦截 `final` 方法（无法被子类重写）或 `static` 方法。
8.  **多线程调用：**
    *   Spring 的事务是通过 `ThreadLocal` 与线程绑定的。如果你在事务方法内开启了新线程执行数据库操作，新线程无法共享主线程的事务。

### 总结建议
*   永远记得在 `@Transactional` 中显式指定 `rollbackFor = Exception.class`。
*   避免类内部方法互相调用事务方法。
*   确保数据库表支持事务。
*   确保异常能够正常抛出到代理层。
