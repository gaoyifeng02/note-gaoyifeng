1. 系统概述

1.1 AIOServer 在体系中的定位

AIOServer 是策略分发与终端管理的中枢服务，位于策略管理层（aiostrategytplsvr / show-strategy-svc）和终端客户端之间，承担策略缓存、分发、匹配、推送任务编排等职责。

2. 策略模型

2.1 三层分层结构

    Module (5种)
      └─ Group (16+种)
          └─ Strategy (70+种)

每层都实现了统一的 Handler 接口，形成处理链：

    // Module 层
    type ModuleHandler interface {
        GetModuleName() string
        GetChangeStrategyList(ctx, req, arg) ModuleRsp
    }
    
    // Group 层
    type GroupHandler interface {
        GetGroupName() string
        GetChangeStrategyList(ctx, req, arg) GroupRsp
    }
    
    // Strategy 层
    type StrategyHandler interface {
        GetStrategyName() string
        AllowAnonymousPull() bool
        GetChangeStrategy(ctx, req, arg) StrategyRsp
    }

2.2 五大模块及其职责

  模块              	名称   	职责                      	典型策略                                    
  BasicModule     	基础对象 	日程、IP组、进程、文件类型等基础数据     	Schedule, IpAddress, Process, FileType  
  CommonModule    	公共策略 	认证、客户端配置、软件管理、合规环境      	Auth, AssetType, DeployConfig, TrustArea, SoftwareControl
  XDLPModule      	数据防泄漏	DLP 策略（泄密审计/管控、外设、端口、水印）	LeakAudit, LeakCtrl, PeriphCtrl, PortCtrl, Watermark, LocalScan
  ComplianceModule	合规策略 	终端合规检查                  	AntivirusCheck, OsCheck, ProcessCheck, LoginDomainCheck
  SWGModule       	上网管控 	安全 Web 网关相关             	Rules, GfwRules, DNS, POP, Advanced, Forward

2.3 gRPC 协议结构

请求： 客户端携带每个策略的当前版本号

    StrategyPullReq
      └─ ModuleReq[]
          └─ GroupReq[]
              └─ StrategyReq { strategyName, version }

响应： 仅返回有变化的策略内容

    StrategyPullRsp
      └─ ModuleRsp[]
          └─ GroupRsp[]
              └─ StrategyRsp { code, strategyName, version, content (Struct) }

2.4 版本管理结构

    type VersionVal struct {
        Ver            int64         // 策略版本号
        StoreTimeStamp int64         // 存储时间戳
        TTL            time.Duration // 有效期（>0 过期，<0 永不过期）
    }

版本号是策略变更检测的核心：客户端每次请求携带上次版本号，服务端比较后决定是否返回新内容。

2.5 Scope 模型（策略适用范围）

    StrategyScope {
        accountId         // 租户
        strategyName      // 策略名
        conditionType[]   // 匹配条件类型
    
        selectGroupUser[] // 包含的用户/用户组
        excludeGroupUser[]// 排除的用户/用户组
        selectSecGroup[]  // 安全组
        terminal_label[]  // 终端标签
        terminal_device[] // 终端设备（机器ID）
        asset_type[]      // 资产类型
    }

Scope 支持本地用户、安全组、资产类型、终端标签、终端设备等多维度，并支持多目录（LDAP/AD）集成。

2.6 动态参数传递

请求在分层处理链中逐层传递上下文：

    GlobalArg {
        Anonymous        bool       // 是否匿名访问
        TotalStrategySize int64     // 累计策略大小
        CompleteAll      bool       // 是否补全缺失策略
        ClientIpInt      int        // 客户端 IP
        OSType           string     // 操作系统类型
        MatchArg         *PolicyMatchArgs  // 策略匹配参数
    }
      └─ ModuleDynamicArg { ModuleName }
          └─ GroupDynamicArg
              └─ StrategyDynamicArg { StrategyVersionMap }

3. 策略下发

3.1 变更通知流程（推模式）

当策略在管理端发生变更时，通过 RabbitMQ 消息驱动更新：

    管理端修改策略 → aiostrategytplsvr 发送 MQ 消息
      ↓
    AIOServer MsgHandler.PreHandle() 校验版本 > 0
      ↓
    MsgParser() 反序列化消息体
      ↓
    SvcCtxGetter() 构建 ServiceContext（租户/用户/客户端维度）
      ↓
    ContentGetter() 获取最新策略内容
      ↓
    UpdateStrategyByCAS() 写入 Redis（乐观锁事务）
      ↓
    Broadcast() 广播通知所有 Pod 过期本地缓存
      ↓
    DefaultMsgErrHandle() 失败时写入 MongoDB（7天 TTL）

系统注册了 95+ 种信号类型到对应的 MsgHandler，覆盖所有策略变更场景。

3.2 客户端拉取流程（拉模式）

终端通过心跳/定时轮询拉取策略：

    Client Request(Module + Group + Strategy + 各策略当前 version)
      ↓
    GetModuleHandler(moduleName) 查找模块处理器
      ↓
    GroupHandler.GetChangeStrategyList()
      ↓ 对每个策略:
      Matcher.CheckVersion(clientVersion)
        ├─ 版本未变 → 返回 nil（无更新）
        ├─ 版本已变且未过期 → LoadContentFromCache() 返回缓存内容
        └─ 版本已变且已过期 → RefreshFromProvider() 从上游 gRPC 回源
      ↓
    返回 StrategyRsp { newVersion, content }

请求间隔控制：

- 默认成功间隔：60 秒（StrategyConfig.ReqInterval）
- 自定义演示场景：10 秒（StrategyConfig.CustomReqInterval）
- 最大响应体大小：500 MB（StrategyConfig.MaxContentSize）

3.3 全量推送机制（PushTask）

用于软件分发、卸载等需要主动推送到大量终端的场景。

3.3.1 两级任务模型

    PushTask（父任务，1个）
      ├─ 定义 Scope（谁需要接收）
      ├─ 定义 Schedule（什么时间窗口执行）
      ├─ 定义 PushFrequency（每分钟调度多少个终端）
      └─ TermPushTask（子任务，每终端1个，最多 20,000）
          └─ 状态机: pending → wait → running → success/fail

3.3.2 终端发现与分批枚举

    按 Scope 条件查询匹配终端:
      ├─ 每批查询: 1,000 个终端
      ├─ 最大循环: 500 次（理论上限 50 万）
      ├─ 单任务硬上限: 20,000 个子任务
      └─ 每批: 过滤兼容性（OS、状态） → InsertMany() 批量写入 MongoDB

3.3.3 三级配额限流

    实际调度数 = min(全局配额, 租户配额, 任务频率配额)

  级别  	配置项                 	默认值 	作用             
  全局  	PushTask.GlobalLimit	2048	全系统并发终端任务总数上限  
  租户  	PushTask.TenantLimit	256 	单租户并发上限，防止资源抢占 
  任务  	PushFrequency（每任务可配）	—   	单个推送任务每分钟可调度终端数

例如 20,000 个终端，PushFrequency=1000，则约 20 分钟分批调度完毕。

3.3.4 调度器分批处理

调度器 (scheduler.go) 每次运行：

    批大小: 256 个终端任务
    最大循环: 32 次（单次最多 256 × 32 = 8,192 个）
    
    流程:
    1. 从 Redis 缓存取 schedulable task IDs
    2. 每次取 256 个，查询终端 + 任务信息
    3. 检查三级配额剩余量
    4. 将符合条件的任务状态改为 "wait"
    5. 配额用完则停止本轮，等下一轮

3.3.5 异步拉取模型

服务端不主动推送到终端，而是标记任务状态为 "wait"，终端通过心跳/prepull 发现并拉取：

    Server: 标记任务状态为 "wait"
      → Terminal: 心跳/prepull 发现待执行任务
      → Terminal: 拉取任务内容并执行
      → Terminal: 上报执行结果
      → Server: 更新任务状态 (success/fail)

3.3.6 推送任务重试

  机制     	说明                   
  重试间隔   	RetryIntervalSecs，可配置
  最大重试次数 	MaxRetryCount，防止无限重试 
  任务超时   	默认 180s，超时自动标记失败     
  失败查询批大小	256 个/批              

3.4 灰度发布

灰度发布通过 AppMode 缓存隔离 + GrayDeliveryProcessor 消息路由 实现。

3.4.1 缓存 Key 隔离

不同版本的 AIOServer 通过 config.AppMode 在 Redis 中维护独立的缓存：

    Global 层: {aio_server:APP_MODE:strategy:global}:version_map
      → 不同 AppMode 的服务实例读写各自的缓存
    
    Tenant/User 层: 共享缓存（不区分灰度版本）
    
    Client 层: 默认支持灰度隔离，可通过 NoGray 标记关闭

3.4.2 NoGray 标记

    // 去除灰度能力（慎用！）
    // 去除灰度的策略缓存会被所有灰度版本共享，需做好高低版本兼容
    func NewClientCacheMetaGeneratorWithNoGray(metas ...string) *ClientCacheMetaGenerator

使用 NoGray 的策略必须保证跨版本兼容性。

3.4.3 消息消费灰度

所有策略消费者都被 GrayDeliveryProcessor 包装：

    normalProcessor := tpl.NewEventMsgDeliveryProcessor(handlers, queue)
    grayProcessor, _ := tpl.NewGrayDeliveryProcessor(normalProcessor, consumerName)

灰度处理器根据消息中的灰度标记，决定当前服务实例是否处理该消息。

4. 策略缓存

4.1 三级缓存架构

    请求 → L1 本地缓存（BigCache，内存）
           ├─ 命中且未过期 → 直接返回
           └─ 未命中/过期 → L2 Redis 缓存（Redis Cluster）
                             ├─ 命中且未过期 → 返回并回写 L1
                             └─ 未命中/过期 → L3 gRPC 回源（上游服务）
                                               └─ 返回并回写 L2 + L1

  级别            	实现                            	特点                              
  L1 本地缓存       	BigCache (allegro/bigcache/v3)	1024 分片, 24h 生命周期, 1GB 硬上限, 线程安全
  L2 Redis 缓存   	Redis Cluster (Hash + String) 	版本与内容分离存储, CAS 事务保证一致性          
  L3 上游 Provider	gRPC 远程调用                     	缓存全部未命中时回源到 show-strategy-svc 等 

4.2 版本与内容分离存储

    版本 Key: {aio_server:strategy:TYPE:ID}:version_map  (Redis Hash)
      └─ Field: strategy_name → Value: VersionVal{Ver, StoreTimeStamp, TTL}
    
    内容 Key: {aio_server:strategy:TYPE:ID}:strategy_name (Redis String)
      └─ Value: 策略内容 JSON

设计要点：

- Hash Tag {}：强制版本 Key 和内容 Key 在同一个 Redis Cluster Slot，支持 WATCH 跨 Key 事务
- Hash 存储版本：一个 Key 存多个策略的版本号，避免 Key 爆炸
- String 存储内容：独立 Key 存内容，避免 Hash 大 Value 问题

4.3 分层 TTL 策略（防缓存雪崩）

  层级    	版本 TTL               	内容 TTL               	说明          
  Global	7天                   	7天                   	全局配置，变化频率低  
  Tenant	24h + random(0-60min)	23h                  	随机偏移防止同时过期  
  User  	3天                   	30min (默认) / 11h (匹配)	用户级粒度细，TTL 短
  Client	3天 + random(0-60min) 	10min                	客户端级，最细粒度   

版本 Map 统一 15 天 TTL：设计上考虑了全公司放假场景（如春节 7 天），避免假期回来后所有缓存同时失效导致雪崩。

4.4 缓存 Key 生成器

四种 CacheMetaGenerator 对应四个缓存层级：

  Generator               	LocalCache Key                       	Redis Key 示例                            	用途                 
  GlobalCacheMetaGenerator	meta_key                             	{aio_server:prod:strategy:global}:version_map	POP、Operation 等全局配置
  TenantCacheMetaGenerator	domain:meta_key                      	{aio_server:strategy:tenant:domain}:version_map	租户级策略              
  UserCacheMetaGenerator  	domain_dir_account:meta_key          	{aio_server:strategy:user:domain_dir_account}:version_map	用户级策略              
  ClientCacheMetaGenerator	domain_machineId_dir_account:meta_key	{aio_server:strategy:client:domain_mid_dir_acc}:version_map	客户端级策略

5. 策略过期与失效
5.1 被动过期（TTL）

各层级按 TTL 自动过期。VersionVal 中存储 StoreTimeStamp + TTL，精确控制过期时间：

    // 判断是否过期
    expired := time.Now().Unix() - v.StoreTimeStamp > int64(v.TTL.Seconds())

5.2 主动过期（Event-Driven）

当策略变更时，通过 4 种 EventHandler 主动清除缓存：

  Handler              	触发条件                          	动作       	场景       
  ExpireByKey          	ExpirationCacheByKey          	删除精确 Key 	单个策略版本变更 
  ExpireByPattern      	ExpirationCacheByPattern      	正则匹配批量删除 	租户下所有策略变更
  ExpireMachineID      	ExpirationMachineIDCacheByKey 	删除后触发重新加载	终端信息变更   
  ExpireLabelByTenantId	ExpirationLabelCacheByTenantId	按租户刷新标签缓存	终端标签变更   

5.3 多 Pod 广播同步（StatusSync）

AIOServer 多实例部署时，本地缓存通过广播机制保持一致：

    策略变更 → Redis 更新完成
      → GlobalSync().Broadcast(ctx, ExpirationEvent)
      → 所有 Pod 通过 StatusSync 通道接收事件
      → EventHandler.Handle() 清除本地缓存
      → 下次请求触发从 Redis 重新加载

5.4 通道重建时的全量清空

当 StatusSync 通信通道重建（如网络故障恢复后），触发 OnChannelCreate 回调：

    // 通道重建时全量清空本地缓存，保证一致性
    localcache.Reset(context.Background())

这是一种保守但安全的策略，确保网络分区恢复后不会出现脏数据。

6. 并发控制

6.1 Redis 层 — 乐观锁 CAS

使用 Redis WATCH/MULTI/EXEC 实现 Compare-And-Swap：

    1. WATCH version_key, content_key       ← 乐观锁监控
    2. GET 当前版本号
    3. 比较: 新版本 > 旧版本?
       ├─ 否 → 返回 OutdatedVersionError（不更新）
       └─ 是 → 执行事务:
           MULTI
             SET content_key → content      ← 先写内容
             HSet version_key → version     ← 再写版本
             EXPIRE version_key → 15天
           EXEC
    4. 如果 TxFailedErr（被其他事务抢占）→ 重试
    5. 最多重试 100 次

关键设计：先写内容、再写版本，保证读取方不会读到新版本号但旧内容。

6.2 本地层 — 细粒度 Key 锁

    type KeyLock struct {
        mutex sync.Mutex                    // 保护 locks map
        locks map[interface{}]*LockObj      // key → 锁对象
    }
    
    type LockObj struct {
        Lock   *sync.Mutex
        Holder int64       // 原子引用计数
    }

特性：

- 引用计数池化：refcount 归零时回收 Mutex，避免内存泄漏
- TryLock 非阻塞：获取不到锁直接返回 codes.Aborted，客户端自行重试
- 防缓存击穿：同一个 Key 只有一个 goroutine 执行回源操作，其他请求等待或快速失败

使用示例：

    lockKey := b.GetLockKey(ctx)
    if !localcache.GlobalKeyLock.TryLock(lockKey) {
        return status.Errorf(codes.Aborted, "lock conflict, client should retry")
    }
    defer localcache.GlobalKeyLock.Unlock(lockKey)
    // 执行缓存刷新...

6.3 防缓存击穿与惊群效应

两层防护配合：

1. Key 锁：同一 Key 只有一个 goroutine 回源，其他直接返回 Aborted
2. 版本双检：加锁后再次检查版本（double-check），避免重复回源
3. TryLock 快速失败：不阻塞请求，客户端侧自带重试逻辑

7. 失败处理与可靠性

7.1 各环节失败处理一览

  场景                 	处理方式                               	影响范围             
  MQ 消息解析失败          	跳过消息，记录日志，写入 MongoDB               	单条消息丢失           
  ServiceContext 构建失败	无效租户/账户，记录日志                       	单条消息             
  Redis CAS 冲突       	自动重试最多 100 次                       	短暂延迟             
  Redis CAS 版本过旧     	返回 OutdatedVersionError，不更新        	无影响（正常现象）        
  TryLock 失败         	返回 codes.Aborted，客户端重试             	单次请求             
  MsgHandler 超时      	ConsumerTimeOut 控制，RabbitMQ requeue	消息延迟处理           
  广播失败               	日志警告，best-effort 投递                	本地缓存短暂不一致（TTL 兜底）
  推送任务执行失败           	按 RetryIntervalSecs 延迟重试           	单终端延迟            
  推送任务超时             	自动标记失败，过期监控回收                      	单终端

8.2 位图数据结构

    type Bitmap struct {
        data []uint64  // 每个 uint64 存储 64 位
    }

  操作             	说明  	实现                                     
  Add(index)     	设置位 	data[index/64] \|= 1 << (index%64)     
  Contains(index)	检查位 	data[index/64] & (1 << (index%64)) != 0
  And(x2)        	交集  	按 chunk 逐个 AND                         
  Or(x2)         	并集  	按 chunk 逐个 OR，动态扩展                     
  AndNot(x2)     	差集  	data[i] &= ^x2.data[i]                 

内存效率： 1,000 个策略 = 16 个 uint64 = 128 字节/位图。

8.3 多维度匹配算法

匹配维度与逻辑关系：

  维度           	类别  	多值逻辑     	跨维度逻辑
  用户 (Include) 	A 组 	OR       	—    
  用户组 (Include)	A 组 	OR（含层级展开）	—    
  源 IP         	A 组 	OR       	—    
  安全组          	A 组 	OR       	—    
  终端设备         	A 组 	OR       	—    
  资产类型         	B 组 	—        	AND  
  终端类型         	B 组 	—        	AND  
  分支           	B 组 	—        	AND  
  终端标签         	B 组 	OR → AND 	AND  
  用户 (Exclude) 	C 组 	OR       	—    
  用户组 (Exclude)	C 组 	OR       	—    

匹配公式：

    A = user_bitmap | usergroup_bitmap | sourceip_bitmap | secgroup_bitmap | device_bitmap
    B = assettype_bitmap & terminaltype_bitmap & branch_bitmap & label_bitmap
    C = excluded_user_bitmap | excluded_usergroup_bitmap
    
    Result = (A & B) & ~C
    

执行步骤：

1. 构建 A 组（OR 合并）：用户包含 + 用户组包含 + 源 IP + 安全组 + 终端设备
2. 构建 B 组（AND 合并）：资产类型 & 终端类型 & 分支 & 终端标签
3. 构建 C 组（OR 合并）：用户排除 + 用户组排除
4. 最终计算：(A & B) & ~C
5. 结果转换：位图 → 策略索引数组 → PolicyMgr 查找策略 ID

8.4 策略解析（PolicyUser 构建）

策略解析将 Scope 定义转换为位图索引：

    对每个策略:
      1. PolicyMgr.Add(policy) → 获得 index (0, 1, 2, ...)
      2. 解析 Scope.Include → 用户组层级展开(DFS) → 写入 LocalOrgMgr
      3. 解析 Scope.Exclude → 同上，写入排除位图
      4. 解析 Scope.SourceIp → IP 范围规则
      5. 解析 Scope.SecurityGroup → 安全组位图
      6. 解析 Scope.AssetType → 资产类型位图
      7. 解析 Scope.TerminalLabel → 终端标签位图
      8. 解析 Scope.TerminalDevice → 终端设备位图
    

8.5 SelectAll 特殊标记

  维度           	SelectAll 标记	含义      
  UserGroup    	"/"         	匹配所有用户  
  TerminalLabel	"all"       	匹配所有标签  
  SecurityGroup	"all"       	匹配所有安全组 
  AssetType    	"all"       	匹配所有资产类型

8.6 BitMapMatcher 缓存管理

BitMapMatcher 管理四层缓存：

  缓存           	级别    	内容             	版本  
  PolicyUser   	Tenant	Scope 定义 → 位图索引	v2  
  UserGroup    	Tenant	用户组层级结构        	v4  
  Bitmap       	Tenant	预计算的匹配位图       	v3  
  UserPolicySet	User  	用户匹配结果         	v1  

重计算触发条件： v1 < max(v2, v3, v4) 时，需要重新计算用户的策略匹配结果。

9. 外部同步

9.1 全量同步 vs 增量同步

  特性  	全量同步         	增量同步         
  触发方式	管理端手动触发或定时   	MQ 消息实时触发    
  数据范围	所有终端数据       	单个/批量终端变更    
  实现方式	临时表比对 + 批量操作 	Merger 策略合并  
  支持场景	初始化、数据修复     	日常运行         
  并发控制	分布式锁（每租户一个任务）	增量期间检查全量是否进行中

9.2 全量同步流程

    Phase 1: 任务创建
      → 分布式锁防止并发
      → 创建 SyncTaskProfile
      → 设置 Redis 同步标记
    
    Phase 2: 数据收集
      → 客户端分批发送终端数据（SyncTerminalReq）
      → 写入临时表（NextBatchToken 分页）
    
    Phase 3: 比对与刷新（CompareAndFlush）
      → 加载临时表 + 生产表数据（各 1,000/批，游标分页）
      → 按 external_id 匹配，CRC 比对变更
      → 分类: New / Edit / Delete / NoChange
      → 批量执行操作（1,000/批，批间休眠 100ms）
      → 递归处理子对象
    
    Phase 4: 清理
      → 删除 Redis 同步标记
      → 删除临时表
      → 更新任务状态
    

9.3 变更检测机制（ChangeDetector）

使用 32 位位掩码 进行字段级变更追踪：

    type ChangeDetector struct {
        fields []string  // 监控的字段列表（最多 32 个）
    }
    
    type ChangeRecord struct {
        ID         string
        ChangeMask uint32  // 每个 bit 代表一个字段是否变更
    }

- 第 0 位 = 第 0 个字段，第 1 位 = 第 1 个字段...
- 通过 reflect.DeepEqual() 比较新旧值
- ChangeMask 允许下游精确知道哪些字段发生了变更

9.4 增量同步（Merger 模式）

根据租户类型选择不同的 Merger 策略：

  Merger           	适用租户     	策略                          
  ZtnaMerger       	ZTNA 租户  	简单覆盖，版本转换（atrust → ztna 模块名）
  LocalAtrustMerger	aTrust 租户	智能合并，保留 AIO 特有字段（策略、用户配置）   

aTrust Merger 特殊处理：

- 跟踪已卸载终端（2 分钟宽限期）
- 合并 SDPC 数据到现有 AIO 终端记录
- 仅当 AIONeedUpdate 标记为 true 时更新

9.5 容错机制

  机制  	说明                                      
  分布式锁	Redis 锁防止并发全量同步                         
  心跳检测	5 秒间隔心跳，DFX 检测卡死任务并重启                   
  超时控制	TaskTimeout 配置（默认 120 分钟）               
  状态机 	Syncing → Interrupt / Finish / Timeout / Terminated
